# Provenance Guard

Provenance Guard is a verification pipeline and API built using Flask. It evaluates text-based content to determine whether it was generated by an artificial intelligence model or composed by a human author. The system is designed for handling and communicating uncertainty transparently while protecting human creators as the ultimate engineering goal.

---

### End-to-End Submission Flow
1. **Ingestion & Rate Limiting:** The user payload hits the `POST /submit` endpoint. `Flask-Limiter` extracts the client identifier, verifying the submission falls within permissible transaction thresholds.
2. **Parallel Signal Evaluation:** The text content is handed off to the Pipeline Controller, which distributes it across three distinct evaluation modules asynchronously:
   * **Semantic/Coherence Analyzer (Signal 1):** Connects to the Groq SDK using the `llama-3.3-70b-versatile` engine to evaluate contextual patterns, structural predictability, and common LLM phrasing.
   * **Sentence Length Variance Engine (Signal 2):** Parses structural variation, sentence length variability, and rhythm profiles native to pure Python.
   * **Lexical Diversity Processor (Signal 3):** Determines the distribution balance of unique words relative to total content lengths.
3. **Ensemble Synthesis:** The individual floating-point outputs are evaluated through a weighted mathematical matrix to calculate a single continuous confidence score.
4. **Label Resolution:** The continuous score is filtered through calibrated thresholds to attach a specific plain-language transparency label.
5. **Persistence & Response:** The complete footprint (timestamp, UUID, individual sub-scores, aggregated score, and metadata) is written to a local structured SQLite database with an active state of `"classified"`. The serialized payload is then returned to the client as an HTTP 200 JSON object.

## Detection Signals

Provenance Guard leverages an **Ensemble Detection** configuration utilizing three separate, non-overlapping methodologies to identify structural, semantic, and mathematical properties of submitted text.

### 1. Holistic Semantic & Coherence Analysis ($S_{llm}$)
* **What it Measures:** Evaluates text for thematic predictability, structural hedging, semantic uniformity, and typical model markers (e.g., over-indexing on transitions like *"Furthermore,"* *"It is important to consider,"* or *"In conclusion"*).
* **Why it Differs:** AI language models are probabilistic text calculators that prioritize highly logical, statistically average transitions. Humans frequently introduce abrupt contextual deviations, colloquial interruptions, and non-linear narrative progressions.
* **Blind Spot:** Professional technical specifications, legal contracts, or standardized academic abstracts naturally favor rigid structural logic and standardized transitional markers. Consequently, formal human-authored text can artificially trigger elevated scores on this signal.

### Signal Choice Reasoning
We chose an ensemble of three complementary, non-overlapping signals to analyze different dimensions of the text:
1. **Holistic Semantic & Coherence Analysis ($S_{llm}$):** Examines thematic predictability, structural hedging, and typical transitional tropes. LLMs excel at modeling high-level semantic transitions, which humans often break with narrative jumps.
2. **Sentence Length Variance Heuristic ($S_{slv}$):** Measures structural "burstiness." Humans naturally vary pacing (mixing short, punchy sentences with long, multi-clause structures). AI models naturally default to highly uniform sentence lengths.
3. **Lexical Diversity Heuristic ($S_{ttr}$):** Calculates the Type-Token Ratio (TTR) normalized against length. Humans use hyper-focused localized jargon and idioms, whereas AI distributes standard vocabulary evenly to maximize probabilistic coverage.

### Ensemble Scoring Logic & Thresholds
The continuous composite score is calculated using the following weighted distribution:

$$Score_{combined} = 0.50 \cdot S_{llm} + 0.25 \cdot S_{slv} + 0.25 \cdot S_{ttr}$$

Contextual semantics ($S_{llm}$) carries a 50% weight because it is the most robust holistic indicator of AI generation. The heuristics carry 25% each to act as structural validation. 

False positives (labeling a human author's original work as AI) carry heavy reputational consequences. Therefore, we use an asymmetric calibration mapping to protect creators:
* **0.00 to 0.35 $\rightarrow$ likely_human (Verified Authentic):** Borders are intentionally relaxed to give creators the benefit of the doubt.
* **0.36 to 0.70 $\rightarrow$ uncertain (Stylistically Mixed):** Contradictory signal combinations (e.g., high semantic structure but highly irregular rhythm) are routed to a neutral "uncertain" state rather than forcing an inaccurate guess.
* **0.71 to 1.00 $\rightarrow$ likely_ai (Automated Patterns Detected):** Triggered only when semantic and structural signals consistently indicate AI patterns.

### Real-World Deployment Changes
If this pipeline were deployed in a real-world production environment, we would implement the following critical improvements:
1. **Asynchronous Processing (Background Queue):** Calling external LLM APIs (like Groq) synchronously inside the request-response loop blocks the thread, move signal computations to an asynchronous task worker then notify the user via WebSockets or polling.
2. **Production-Grade Database:** Replace the local file-based SQLite database with a production database like PostgreSQL to handle concurrent database transactions, connection pooling, and replication.
3. **Linguistic Caching Layer:** Implement a Redis caching layer based on the SHA-256 hashes of submitted text blocks. If the exact same text is submitted multiple times, it will return the cached evaluation, avoiding redundant LLM API calls and costs.
4. **Self-Hosted Model:** Fine-tune a smaller local classifier model (e.g., DeBERTa-v3 or a distilled RoBERTa) to replace the Groq API call. This would decrease network latency, eliminate dependency on external API keys, and drastically reduce per-request compute costs.

## Transparency Labels

The transparency labels completely omit engineering jargon, translating diagnostics into clear, actionable outcomes. The exact text returned and displayed for each variant is:

### 1. High-Confidence Human (Score Range: 0.00 – 0.35)
* **Title:** `Verified Authentic`
* **Body:** `This text displays the natural variations in rhythm, pacing, and vocabulary choices characteristic of direct human writing.`

### 2. Uncertain / Ambiguous (Score Range: 0.36 – 0.70)
* **Title:** `Stylistically Mixed`
* **Body:** `This text includes a blend of patterns. It features highly organized structures common in automated assistants alongside natural variations typical of personal expression.`

### 3. High-Confidence AI (Score Range: 0.71 – 1.00)
* **Title:** `Automated Patterns Detected`
* **Body:** `This text closely matches the uniform structure, repeating phrase links, and predictable sentence spacing typically generated by AI language models.`

## Operational Guardrails & Rate Limiting

### Policy Configuration
The `POST /submit` endpoint is hardened using `Flask-Limiter` with an explicit dual-window operational ceiling:
* **10 requests per minute**
* **100 requests per day**

### Technical & Systemic Rationale
1. **Human Pacing Alignment:** Organic human writers do not publish long-form essays or text excerpts every few seconds. A ceiling of 10 checks per minute accommodates standard content revisions, copying/pasting edits, or multi-chapter reviews without interrupting legitimate author workflows.
2. **Abuse & Scraping Mitigation:** Malicious actors attempting to reverse-engineer our detection thresholds or run automated scripts to optimize spam bots will trigger the 10-request block rate limiter. This protects downstream resources and prevents system degradation.
3. **Graceful Failure State:** When a transaction exceeds these boundaries, the server returns an HTTP 429 Too Many Requests status code with a structured JSON error body, preventing thread exhaustion.

## Appeals Workflow & Moderator Experience

When an author contests an automated classification, Provenance Guard provides a transparent path for human review rather than relying on automated re-classification.

### Core Appeal Logic
* **Endpoint:** `POST /appeal`
* **Required Parameters:** `content_id` (UUIDv4 matches submission records), `creator_reasoning` (Text block explaining context).
* **Database State Mutability:** Executes an atomic update changing the submission's database status from `"classified"` to `"under_review"`. The `creator_reasoning` string is securely appended to that record row within the permanent SQLite engine.

## Known Limitations

### 1. Non-Native English Academic Writing Styles
* **Linguistic Cause:** Human authors who learn English in formal academic environments are taught using rigid structural templates and transition formulas (e.g., *"Furthermore, it must be stated,"* *"Consequently, the data suggests"*).
* **Pipeline Failure Mechanism:** This high structural rigidity directly fools both the Sentence Length Variance Heuristic (registering low variance/high uniformity) and the Groq Semantic Analyzer (identifying academic transitional tokens as common LLM phrasing), incorrectly triggering a high AI score.

### 2. Minimalist Copywriting and Modern Poetry
* **Linguistic Cause:** Professional sales copy, landing page catchphrases, or minimalist poetry use highly repetitive vocabulary, uniform phrase lengths, and short, evenly spaced sentences for dramatic effect.
* **Pipeline Failure Mechanism:** The low Type-Token Ratio (due to intentional repetition) combined with zero sentence length variance (rhythmic uniformity) will fool both structural heuristics, resulting in an artificially elevated AI score for a highly curated human piece.

## Spec Reflection

* Documenting our structural layouts in `planning.md` before coding prevented design drift. It established that the continuous scoring model shouldn't just act as a binary switch at `0.5`. Instead treat the middle zone (`0.36 - 0.70`) as an entirely independent, non-committal UX path (`uncertain`), saving human authors from false positive flags.
* **How We Diverged:** During testing of the four benchmark inputs, the initial linear variance model was too sensitive; short texts naturally have low sentence length variance, causing them to fall into the "Uncertain" category purely due to word count limitations. To resolve this, we:
  1. Added a length-normalization scaling factor to smooth out short text bodies.
  2. Adjusted the stylometric heuristics sensitivities, setting the variance scale factor to `50.0` and the TTR deviation scale factor to `0.32`. This successfully separated the benchmark inputs, mapping the AI-generated text to `likely_ai` and the human-written text to `likely_human`.
